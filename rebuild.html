02 Run 0 — Architecture Rules

These rules are immutable. If you want to change the architecture, write a new versioned contract document first. Never modify existing code to break the contract.

🚫
These are not guidelines. They are system law. Any code that violates them is invalid architecture, regardless of whether it works.
#RuleWhat it forbids
01
Single Source of Truth
One global state store in storage.js. No duplicates.
local component state as SSOT
02
Event-First Architecture
Everything starts as an event. State changes only after event processing.
direct state mutation
03
Storage Abstraction
All storage through the adapter interface.
calling localStorage or DB directly
04
Module Isolation
Modules communicate only via events/actions.
cross-module direct writes
05
AI Non-Destructive
AI suggests and analyses, never mutates.
AI bypassing action engine
06
Rule Engine Authority
Rules evaluated before every action.
skipping rule evaluation
07
UI Read-Only
UI dispatches events and renders state only.
UI direct state mutation
08
No Hard Coupling
No direct storage calls outside adapter, no hidden deps.
cross-layer hidden imports

03 Layer Dependency Rules

Higher layers import from lower ones. Lower layers must never import from higher ones. This is the dependency direction — never reverse it.

governance/← imports fromagents/, saas/, auth/, core/, nocode/
agents/← imports fromcore/, ai/, saas/, ui/, nocode/
nocode/← imports fromcore/, saas/, ui/
ecosystem/← imports fromcore/, saas/
saas/← imports fromcore/, auth/
ai/← imports fromcore/
ui/← imports fromcore/, ai/
auth/← imports fromcore/
core/← imports from(nothing — foundation only)
bash — audit back-references
# Run this after any change — should always return nothing
grep -rn "from.*governance/" bco/core/ bco/ai/ bco/saas/
grep -rn "from.*agents/" bco/core/ bco/ai/ bco/ui/ bco/saas/

04 Universal Event & Action Format

Every event and action in the system must match these exact shapes. These formats are fixed by Run 0 — never invent a wrapper or rename a field.

js — Event shape
{
  id:        crypto.randomUUID(),   // always generate fresh
  type:      "ALL_CAPS_SNAKE_CASE",   // string constant
  source:   "user" | "ai" | "system" | "agent" | "workflow",
  module:    "MODULE_NAME",
  payload:   {},                    // arbitrary object
  timestamp: new Date().toISOString()
}
js — Action shape
{
  id:          crypto.randomUUID(),
  type:        "ALL_CAPS_SNAKE_CASE",
  status:      "pending" | "approved" | "rejected" | "executed",
  triggeredBy: "event_id",
  payload:     {},
  timestamp:   new Date().toISOString()
}
ℹ
Never add a data: wrapper. Never use camelCase for the type field. These shapes are validated at the governance layer — wrong format = blocked action.

05 Storage API Contract

The system exposes exactly five storage methods. These are the only ways to read or write state. The adapter is swappable (Local → Supabase → custom) — that is the whole point.

js — The only five methods
StorageAdapter.get(key)
StorageAdapter.set(key, value)
StorageAdapter.update(key, partialValue)
StorageAdapter.delete(key)
StorageAdapter.subscribe(key, callback)
⚠
Never call localStorage, supabase, or any DB directly. If you need a new storage backend, write a new adapter in core/storage-adapter.js — don't bypass it.
js — Key naming convention
// Tenant-scoped keys
`t:${tenantId}:bco_modules`
`t:${tenantId}:bco_workflows`

// Global platform keys (no prefix)
"bco_audit_log"
"bco_autonomy_mode"

06 Adding a New Module

Every module must follow the module contract from Run 0. Use bco/modules/sleep.module.js as a reference implementation.

js — Module contract
export default {
  name:        "MY_MODULE",
  entities:    [],   // data schemas
  actions:     [],   // action types this module handles
  rules:       [],   // rules this module registers
  ui_blocks:   [],   // UI components it provides
  permissions: []    // who can access it
};
✓
Register via registerModule() from core/modules.js. Communicate only via dispatchEvent() and dispatchAction(). Never write directly to another module's state.

07 Adding a New Agent Type

The agent system uses capability gates and a forbidden list (§13) enforced at runtime — not just at creation time.

js — Steps to add an agent type
// 1. Add to AGENT_TYPES in agents/agent-core.js
export const AGENT_TYPES = {
  // ... existing types ...
  MY_AGENT: "my_agent"
};

// 2. Add capabilities to AGENT_CAPABILITIES
AGENT_CAPABILITIES[AGENT_TYPES.MY_AGENT] = [
  "do_safe_thing",
  "analyse_data"
];

// 3. Confirm NONE of these appear in AGENT_FORBIDDEN
// _assertNotForbidden() will throw at execution time if they do
🚫
§13 is absolute. Never give an agent direct storage access. All state mutations go through dispatchAction(). The forbidden list covers: mutate_billing, mutate_permissions, bypass_rule_engine, cross_tenant_read, delete_tenant.

08 Adding a Governance Check

governance-engine.js runs five checks in strict order: tenant → permission → billing → safety → autonomy. Adding a sixth is straightforward.

js — Add a 6th governance check
// 1. Write the check function
function _checkYourThing(action, context) {
  if (somethingBad) {
    return { ok: false, reason: "Explain why it was blocked." };
  }
  return { ok: true, reason: "" };
}

// 2. Call it inside governanceCheck()
const yourResult = _checkYourThing(action, context);
if (!yourResult.ok) violations.push("your_check");

// 3. Include in allowed condition
const allowed = violations.length === 0;

// Every action is still audit-logged — do not remove that.

09 Adding a Workflow Node

There are 20 node types across 5 categories. Adding a new one requires changes in 3 files.

js — 3-file checklist for a new node
// File 1: nocode/workflow-schema.js
NODE_TYPES.MY_NODE = "MY_NODE";
NODE_CATEGORIES.action.push(NODE_TYPES.MY_NODE); // or other category

// File 2: nocode/workflow-engine.js — _executeActionNode()
case NODE_TYPES.MY_NODE:
  return await _handleMyNode(node, context);

// File 3: nocode/workflow-templates.js (optional but recommended)
// Add a template entry if this is a common pattern
ℹ
If the new node is a trigger type, also add it to evaluateTrigger() in workflow-engine.js.

10 Runtime Notes

⚙
No build step required. BCO Core is plain ES modules. Bundle with Vite, Rollup, or esbuild for browser use. Set "type": "module" in package.json.
TopicDetail
Module syntaxAll files use import/export. No CommonJS require().
RuntimesModern browsers (bundled), Deno, Node 18+, Bun
crypto.randomUUID()Used throughout. Native in modern browsers and Node 15+.
Sandboxecosystem/sandbox.js uses Web Worker in browsers, node:vm in Node. Ensure your runtime supports the target.
performance.memoryagents/metrics.js — browser only. Returns -1 in Node. Expected, not a bug.
In-tab reactivitystorage.subscribe() is cross-tab via StorageEvent. For same-tab reactivity add a custom EventEmitter (future Run 11).
Tenant isolationAll tenant data uses t:{tenantId}: key prefix. Never mix tenant keys.
